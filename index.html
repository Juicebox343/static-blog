Podcast Player

1/27/20

I keep writing myself down this tunnel despite knowing.. like.. knowing in my bones.. That there is a better way to do this.

For now I just want something done. And it nearly is.

The player itself is currently here: Github - player.js A lot of that code is thanks to MDN’s guidance.

For the moment, my feed is fetched and parsed, pulling Title, URL and Description into their own variables for each item in the feed.

Also in that loop, the makeAPlayer function is called, which adds a bunch of HTML to the document, inserting those variables when needed, plus an episode counter that gets tacked onto the end of IDs for each episode found in the feed.

I have to say, it was simultaneously thrilling and humbling to realize I didn't need to individually create each HTML element and then attach it to its parent. Thrilling because it was so much cleaner to just change the innerText of a single element. Humbling because I'd already done all the work of typing out and attaching each element manually. :(

Still in that function, all the event listeners are added according to the IDs + the epCount.

And then down the road some math is done to get the timing to translate into something more readable.

This feels only slightly better than spaghetti code.

In my head the player itself should be an object that has its own methods. And then I replicate it and pass things into it to get myself a page full of separate audio players.

As it stands now, though. It works. So I’m going to move on to generally completing the Tandem page and then circle back to see what can be done with the audio player.

Fetch()ing Feeds

1/20/20

Work on the podcast player is slow.

I’ve got the main functionality of it set up as far as I can tell. Play/Pause works, Mute/Unmute, I can click the progress bar to change where I am in the file, I can change the volume, and the time comes up once the metadata has loaded.

Title info is hardcoded at the moment as I want that to be pulled from the RSS feed, same with the URL.

I’ve mentioned before that my learning started on content based on the JS landscape of 4-5 years ago, so my first instinct for pulling an RSS feed? XMLHttprequest.

I’m sure it would work. I’ve used it for other things, but after awhile I shifted over to fetch(). I’ve never used it and its simpler format helped sort of clear my screen of distractions as I tried to figure out parsing and such.

I’m nearly there. I can pull all the information I want from the feed, but now.. Now I need to create a player for each item pulled from the XML.

I thought this would be easier than it is, and I’m confident it should be much easier than I’m making it. Something with classes? I don’t anything about classes but I assume they’re the solution to all of my problems.

For the moment I’m setting up JS to dynamically create the HTML elements I need for my player, setting attributes of each element, and adjusting the ID of each to be something like.. “Player-controls” + itemNum

So that I end up with unique IDs. In my head this means the play/pause button and title information and all that will point to the right place. But this feels super sloppy and sort of.. Unconfirmed. We’ll see how it works once I’m done.

I’m so confident there’s a better way to do this. I’m just not sure how to search for it yet.

Development Courses

1/16/20

My education has been a bit spotty. I’ve taken C++ classes in high school, done Python tutorials for the Blender Game Engine, and had several false starts in the area of web development.

My first class was based on a podcast ad I heard enough times that I decided to go for it. OneMonth. The class itself was fine if superficial. It characterizes itself as prepping you for a job in something like 15 minutes a day for a month.

After a month I can tell you I did not feel ready to tackle original problems, but I did churn out several small projects very quickly, which was enough to keep the original of interest spark lit.

I moved from OneMonth and found Udemy. Through much poking around I eventually landed on Colt Steele’s Web Developer Bootcamp. This ended up being exactly the kind of guided learning I needed, and Colt seemed to know what he was doing.

Halfway through the course, I realized it was created 4 years before I had started it, and in those 4 years the changes brought in by ES6 were taking root.

The Javascript I was learning was not invalid per se, but it wasn’t up to date. It wasn’t as clean as it could be now, and because of the nature of a scattershot course like that that intends to capture all aspects of Full Stack development, my foundational JS was pretty shallow anyway.

I’ve been spinning my wheels for several months after having completed that course. Attempting to take what I’ve learned and apply it to my own projects, while also finding that syntax on StackOverflow or MDN is not quite what I’m familiar with.

This caused a number of mental blocks and countless hours of retracing my code, trying to split my learning when my grasp on what I had written was tenuous at best.

I should have done this much earlier, but I recently purchased Wes Bos’s Beginner Javascript course and oh boy oh boy. Things are turning around for me.

So many tiny foundational elements that I could have been relying on are coming to the surface. I finally get that in depth discussion and tutelage you can only get through a targeted course.

Zero regrets buying this course. Maybe its basic for old pro’s but for a largely self taught person like me, it’s providing something that was sorely needed.

Digital Ocean and Node.js

1/15/20

After struggling to make any headway with my existing host, A2, in getting my Express based Star Wars RPG app running, I decided to give Digital Ocean a try.

I’m not sure where I heard of Digital Ocean but it seemed like a good alternative to the much recommended Heroku. I’m avoiding Heroku for the moment because it requires that I use PostgreSQL and I’m committed to getting a few functioning MySQL projects completed before I start splitting hairs and dividing my focus too much.

I spent all those hours learning MySQL because it seemed like the best relational database to learn if I were to only learn one. So I’m going to use it.

My assistant app is not in a final form at all, but it’s close enough that I’m ready to try deploying something, and it’s proven to be functional on my local machine. So how hard could it be?

The app itself is written using the NodeJS framework, Express, and MySQL as the database.

Now, on to what happened today.

After clumsily getting everything set up on my end, ensuring my environment variables were in place, etc., I began the process as laid out by the Digital Ocean docs and to be honest? It went swimmingly for such a long time I thought I might actually pull this off.

Keep in mind I’ve been struggling with deployment for a few days, so this was a pretty exciting prospect for me.

Things started to get bumpy here and there as a few of the tutorials specified specific versions of Node or whatever else, thinking version was important for the proper functioning of Digital Ocean, I abided by these things. Turns out I needed to get all that replaced and upgraded to get any of the additional packages working together.

And then something popped up I had never heard of before. Reverse Proxies.

Surely this isn’t for me? Nginx? What is that. I’ve never heard of that so it must be for something more advanced than deploying a simple website. Right?

Well eventually I figured out it is precisely for me. At least if I’m hoping to model realistic development expectations. Express itself suggests using a reverse proxy to actually render your web pages. That sentence I just typed out makes very little sense to me but I’m confident I used the correct words in the correct order.

After pouring over several tutorials both on and off Digital Ocean’s own library of community tutorials, I can’t say I understand it yet.

But through much hammering I managed to get something to appear on my screen. The DNS isn’t resolving for some reason yet. Who knows why. But I can type in the IP of my website and get--some syntax errors?

I haven’t chased down why this is the case yet, but on my local machine formatting my EJS like so:

<% include ../views/header %>

Is just fine.

But as I eventually realized thanks to StackOverflow, the server or the browser or something is looking for this:

<%- include (‘../views/header’) %>

Visually I can see that the second one is a bit nicer to look at. But I was taught the first one, so why doesn’t it work?

I went through every page of my app to correct that and it seems to be working in that regard now.. But then another problem came up.

Something about how I’m getting my information on the ID of the logged in user is pulling up as undefined.
